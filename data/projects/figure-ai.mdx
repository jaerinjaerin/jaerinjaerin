## 🎯 프로젝트 상세

### 1. Fetch API로 인한 초기 로딩 지연을 **DB 직접 접근으로 변경하여** **네트워크 오버헤드 제거**, **초기 렌더링 속도 개선** 및 사용자 경험 향상

- **문제 및 원인**
  - 초기 페이지 로딩 시 서버 컴포넌트에서 내부 API를 fetch로 호출하여 불필요한 네트워크 왕복이 발생
  - 동일 서버 내에서 HTTP 요청을 거치는 구조로 인해 초기 렌더링이 지연됨.
- **해결 과정**

  - 잦은 mutation이 발생하지 않는 정적 데이터의 경우, 서버에서 직접 조회하여 초기 렌더링 시 즉시 제공하는 것이 사용자 경험에 유리하다고 판단함
  - 서버 컴포넌트에서 Fetch API 호출 대신 Prisma를 통해 DB에 직접 접근하는 방식으로 변경

    ```tsx
    import prisma form '../prisma';
    import { getUserLocale } from '@/lib/locale';

    export const getSubsriptionPlans = async() => {
      const locale = await getUserLocale();

      const plans = await prisma.subscriptionPlan.findMany({
        where: {
          isAcive:true,
          OR:[
            { locale: locale },
            { locale: null }
          ],
        },
        orderByd: {
          displayOrder: "asc",
        },
      })

      return plans;
    }
    ```

- **결과**
  - 불필요한 네트워크 홉 제거로 초기 렌더링 속도 개선
  - 서버 리소스 사용 효율화 및 응답 지연 감소로 사용자 체감 성능 향상

### 2. 설문 제출 시 **Google Sheets 자동 저장 + Discord 실시간 알림 기능 구현**

![image.png](/projects/figure-ai/8.png)

- **문제 및 원인**
  - 사용자 피드백 확인을 위해 기획자가 개발자에게 데이터 추출을 요청해야 했음.
  - Google Sheets 연동으로 접근성을 개선했으나, 시트를 직접 열어봐야 하는 구조적 한계로 실시간 대응이 어려웠음.
- **해결 과정**
  - `api/feedback` 엔드포인트를 통해 피드백 데이터를 Google Sheets에 자동 저장하도록 구현
    ```tsx
    try{
    	setLoading(true);
    	const response = await fetch("/api/feddback", {
    		method: "POST",
    		headers: {
    			"Content-Type": "application/json",
    		},
    		body: JSON.stringify(data)
    	})
    } catch(error){ //... }
    ```
  - Discord Webhook을 추가하여 피드백 제출 즉시 알림이 발송되도록 개선, 실시간 모니터링 및 즉각 대응이 가능해짐.
    ```tsx
    const response = await fetch(DISCORD_WEBHOOK_URL, {
      method: 'POST',
      headers: {
        'Content-Type': 'application/json',
      },
      body: JSON.stringify(payload),
    });
    ```
- **결과**
  - 기획자가 스프레드시트에서 직접 데이터 조회/분석 가능 → 데이터 요청 대기 시간 제거
  - 설문 제출 즉시 Discord 알림 발송 → 피드백 인지 시간 단축
  - 개발자의 데이터 추출 업무 제거를 통해 DX 개선 및 팀 간 업무 의존도 감소

### 🎓 배운 점 (Lessons Learned)

1. **URL 설계는 사용자 경험과 직결된다**: Query Parameter와 Path Parameter 중 어떤 방식을 선택할지 고민하며 라우팅 구조를 깊이 이해할 수 있었습니다. 리소스 식별에는 Path(`/style/anime`), 필터링/정렬에는 Query(`?sort=latest`)를 사용하는 기준을 세웠고, URL만 봐도 페이지의 맥락을 파악할 수 있도록 설계하는 것이 사용자 경험에 영향을 미친다는 것을 배웠습니다.

2. **관심사의 분리가 유지보수성을 높인다**: 비즈니스 로직은 Custom Hook에서 처리하고, 컴포넌트는 전달받은 데이터를 렌더링하는 역할만 담당하도록 분리했습니다. 이렇게 "멍청한 컴포넌트(Dumb Component)" 패턴을 적용하니 UI 변경 시 로직을 건드리지 않아도 되어 수정 범위가 명확해지고, 테스트와 재사용이 쉬워졌습니다.
